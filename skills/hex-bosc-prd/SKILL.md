# BOSC 业务需求书 (PRD) 生成器

## 概述

此 Skill 专门用于生成符合 **BOSC (上海银行)** 标准的《业务需求书_业务功能类V4.8》文档。通过结构化的调研和分析，协助产品经理和业务人员产出规范、严谨且可直接用于技术评审的需求文档。

## 核心工作流

当用户要求创建或优化 BOSC 需求书时（例如：“帮我写一个上海银行的反洗钱监控功能需求书”），请遵循以下流程：

### 第一步：收集背景与核心要素 (Gather Context)

在开始生成之前，必须通过对话收集以下关键信息（如果用户已提供部分，则补充缺失项）：

**必须明确的信息：**
- **需求名称**：本次项目的具体名称。
- **业务背景**：核心痛点是什么？为什么要启动这个项目？
- **业务目标**：通过本项目希望达到什么效果？
- **用户群体**：行内人员（哪些部门？）还是行外客户？预计用户数？
- **发起渠道**：PC端、手机银行、网银、柜面等。

**推荐提问清单：**
1. 这项需求的背景和核心痛点是什么？
2. 预期的业务目标和建设目标是什么？
3. 哪些用户会使用这个功能？大概有多少用户？
4. 功能主要在哪个渠道（PC/App/柜面）发起？
5. 是否有明确的系统边界或技术限制？
6. 哪些业务规则（控制逻辑）是必须严格执行的？

### 第二步：生成符合 V4.8 模板的文档结构

使用 `references/bosc_prd_template.md` 作为基准结构，生成包含以下部分的文档：

1. **需求概述**
   - **需求背景**：详细阐述痛点。
   - **需求目标**：明确具体目标。
   - **总体需求描述**：建议配合 Mermaid 流程图或逻辑图。
   - **需求条目清单**：拆分为最小业务单位。

2. **功能详述** (按条目拆分)
   - **用户信息/渠道**。
   - **功能流程图**：使用 Mermaid `graph TD` 或 `sequenceDiagram`。
   - **功能概述**：一句话说明核心逻辑。
   - **控制逻辑**：以表格形式详述流程环节与业务规则。
   - **异常处理/特殊说明**。
   - **后端数据处理**：涵盖流向、采集、脱敏、加密等要求。

3. **界面展示**
   - 描述界面布局或提供线框图描述。

4. **输入输出要素/业务数据要素**
   - 必须包含字段名、类型、长度、输入方式、必输项、默认值、下拉选项。

### 第三步：深度填充控制逻辑与数据要素

这是 BOSC 需求书的关键，务必做到：
- **控制逻辑**：逻辑应严密，区分系统硬控制（如：非空校验、权限检查）和业务软校验。
- **数据要素**：字段长度和精度需符合银行系统规范（如：金额通常为 decimal(18,2)）。

## 专项指南

### 1. 需求条目拆分原则
- 条目应是业务闭环的最小单位。
- 例如：不再将“转账”作为一个条目，而是拆分为“转账录入”、“转账复核”、“转账结果查询”。

### 2. 控制逻辑表格编写范例
| 流程环节 | 业务规则 | 说明 |
| :--- | :--- | :--- |
| 信息输入 | 系统应校验金额大于0且不超过账户可用余额 | 余额检查逻辑 |
| 权限验证 | 仅限总行消保部具有“审核”权限人员可操作 | 角色硬控制 |

### 3. 数据要素字段定义要求
- **输入方式**：下拉选择、文本录入、自动带入、只读。
- **敏感字段**：若涉及手机号、身份证、卡号，必须在“后端数据处理”中明确脱敏策略。

## 使用场景

- **场景 1：全新功能需求**
  - “基于上海银行V4.8模板，为我写一个‘贷款额度试算’功能的PRD。”
- **场景 2：功能优化需求**
  - “优化现有的‘网银转账’功能，增加指纹验证逻辑，请更新BOSC PRD。”
- **场景 3：字段要素梳理**
  - “根据业务流程，帮我列出‘客户准入审核’功能需要的所有输入输出要素表格。”

## 校验与质量标准

产出的文档应通过以下检查：
- [ ] 包含所有带有 [必需] 标签的标题。
- [ ] 逻辑图中不含死循环，逻辑闭环。
- [ ] 数据要素表包含所有关键维度（长度、精度、输入方式等）。
- [ ] 明确了敏感字段的脱敏处理要求。
