# HEX BOSC SRS Generator

## Overview

生成符合上海银行（BOSC）V4.8 标准的高质量业务需求书（SRS）。本 skill 旨在帮助需求分析人员和业务经理创建结构清晰、内容完整且符合银行监管和内控要求的业务需求文档。

## Core Workflow

当用户请求创建 BOSC 业务需求书时（例如：“为智能存款业务创建一个 BOSC 业务需求书”），请遵循以下流程：

### Step 1: Gather Context (获取背景)

在生成文档前，通过发现对话收集核心信息：

**必要信息：**
- **需求名称**：项目或功能名称是什么？
- **业务背景**：为什么要建设这个项目？解决什么现状问题？
- **目标用户**：谁是主要受益者？预计有多少用户？
- **业务目标**：期望达到什么可量化的 KPI 指标？
- **关联系统**：是否涉及会计核算、监管报送或数据应用？

**发现问题示例：**
```
1. 本次需求的具体业务背景和痛点是什么？
2. 目标客群是谁？预计上线后的 MAU 或交易量目标？
3. 是否涉及外部合作方？
4. 是否涉及会计核算、借贷分录或中收计费？
5. 是否有特殊的安全或审计日志要求？
6. 本次需求的紧急程度和预期投产时间？
```

### Step 2: Generate SRS Structure (生成结构)

使用 `references/srs_template.md` 模板创建文档。必须包含以下核心章节：

1. **需求概述** - 背景、目标、价值评估（BOSC 规范要求）。
2. **总体需求描述** - 业务逻辑和差距分析。
3. **需求条目清单** - 拆分为最小可交付单元。
4. **功能详述** - 用户、渠道、控制逻辑、界面要素。
5. **关联影响分析** - 会计、监管、风险、运营等深度分析（银行系统核心）。
6. **非功能性要求** - 服务级别、存储、安全、审计。

### Step 3: Detailed Functional Design (详细功能设计)

对于每个功能条目，采用 BOSC 规范格式：
- **控制逻辑**：明确流程环节、业务规则和异常处理。
- **界面要素**：参考《附件0：输入输出要素模板》，列出字段名、格式、是否必输、控制逻辑。
- **后端处理**：强调数据敏感度等级、脱敏和加密。

### Step 4: Cross-Impact Analysis (关联影响分析)

这是 BOSC 模板的核心部分。必须详细说明：
- **会计核算**：明确核算场景及会计分录。
- **监管风险**：对标 EAST、反洗钱、征信等报送要求。
- **数据应用**：说明对 CRM、UDAS 等下游系统的数据同步要求。

### Step 5: Validate & Review (验证与评审)

运行验证脚本确保文档完整性：
```bash
scripts/validate_srs.sh <srs_file.md>
```

## Usage Patterns (使用模式)

### Pattern 1: 新业务功能需求
**用户请求**：“创建一个‘反诈转账提醒’业务需求书”
**执行**：关注控制逻辑中的异常处理（容错原则）和监管关联影响（反欺诈）。

### Pattern 2: 流程优化需求
**用户请求**：“优化现有的手机银行开户流程”
**执行**：重点描述“现状分析”中的流程断点，并提供优化前后的对比流程。

### Pattern 3: 监管合规需求
**用户请求**：“根据反洗钱新规更新个人客户开户要素”
**执行**：严格对标《附件4：反洗钱数据要求》，并在“监管报送”章节详细列出影响项。

## BOSC 最佳实践

### 体验设计原则
1. **通俗**：贴近场景。
2. **清晰**：主次突出。
3. **简单**：操作应少。
4. **反馈**：及时明确。
5. **容错**：行为包容。

### 价值评估维度
- **业务创利**：存贷增量、利润。
- **业务成本**：获客成本、人力成本。
- **对客体验**：MAU、留存率。
- **内部管理**：流程自动化率、节约时长。
